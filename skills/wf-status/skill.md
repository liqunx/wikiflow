# /wf-status - 显示状态

**⚠️ 重要约束：本技能必须在文档仓库中执行**

显示 WikiFlow 当前状态和任务进度，列出所有活跃开发计划。

**架构依据**：[current/architecture.md](../../docs/wiki/current/architecture.md) - 文档与代码分离

## 使用方式

```bash
# 显示所有活跃开发计划
/wf-status

# 显示详细信息
/wf-status --detail

# 按状态筛选
/wf-status --active
/wf-status --draft

# 关键词筛选
/wf-status website
/wf-status "git"

# 组合筛选
/wf-status --active "website"

# 自然语言筛选
/wf-status "显示包含 website 的计划"
/wf-status "进度超过 50% 的计划"
```

## 执行流程

### 0. 前置检查

```
读取 .wikiflow/config.json，获取路径配置。

**检查当前仓库类型**：

检查当前目录特征：
  IF 存在 .wikiflow/config.json AND docs/wiki/ 目录:
    → 当前是文档仓库
    → ✅ 继续执行

  ELSE:
    → 当前不是文档仓库
    → ❌ 错误：wf-status 必须在文档仓库执行
    → 报错并提示：
      "❌ 操作被拒绝：wf-status 必须在文档仓库执行

       当前仓库：代码仓库（或无法识别）
       尝试操作：查看状态
       允许操作：代码修改、测试、构建

       请切换到文档仓库后重试：
       1. 查找文档仓库目录（如 ../wikiflowDev）
       2. 切换到文档仓库目录
       3. 再次执行 /wf-status"
    → 退出执行

检查 active 目录是否存在：
  → 检查 {paths.changes}/active/ 是否存在
  → 如果不存在，提示用户："当前没有活跃的开发计划"
```

### 1. 扫描 active 目录

```
扫描目录：{paths.changes}/active/

FOR 每个子目录:
  → 验证是否包含 spec.md 和 tasks.md
  → 如果缺少文件，跳过该目录并警告

收集所有有效的开发计划：
  → 目录名（如 2026-04-24-wf-status-command）
  → spec.md 路径
  → tasks.md 路径
  → decisions.md 路径（可选）
```

### 2. 解析计划数据

```
FOR 每个开发计划:
  1. 读取 spec.md：
     → 提取标题（第一行，去掉 # 符号）
     → 提取元数据：创建时间、状态、相关 Wiki
     → 提取目标（## 目标部分的第一段）
     → 提取描述（从目标或当前问题中提取）

  2. 读取 tasks.md：
     → 统计总任务数：匹配 `- [` （所有任务）
     → 统计已完成：匹配 `- [x]`
     → 统计待执行：匹配 `- [ ]`
     → 计算进度百分比：(已完成 / 总数) * 100

  3. 获取最后更新时间：
     → 优先读取 decisions.md 的修改时间
     → 如果 decisions.md 不存在，读取 tasks.md 的修改时间

  4. 处理目录名：
     → 从目录名中提取短名称（去掉日期前缀）
     → 如：2026-04-24-wf-status-command → wf-status-command
```

### 3. 应用筛选条件

```
IF 用户指定了筛选参数:
  → 按状态筛选：--active、--draft、--paused、--blocked
  → 关键词筛选：匹配计划名或描述中的关键词
  → 组合筛选：AND 逻辑（如 --active AND 包含 "website"）

IF 用户使用自然语言:
  → 解析自然语言模式：
    • "显示包含 X 的计划" → 关键词筛选
    • "进度超过 X% 的计划" → 百分比筛选
    • "昨天创建的计划" → 时间筛选
    • "状态为 active 的计划" → 状态筛选

只保留符合条件的计划
```

### 4. 生成输出

```
生成标准化的状态输出：

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

📊 WikiFlow 状态概览

活跃开发计划：{N} 个

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. {计划短名称}
   状态：{active|draft|paused|blocked}
   进度：{进度条} {百分比}% ({已完成}/{总数} 任务完成)
   创建时间：{YYYY-MM-DD} | 最后更新：{YYYY-MM-DD HH:MM}
   描述：{简短描述（最多 50 字）}

2. {计划短名称}
   ...

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

如果用户指定 --detail：
  → 显示完整的任务列表
  → 按模块分组
  → 标注任务状态（✅ 已完成、⏳ 待执行）

如果用户没有指定筛选：
  → 添加"下一步建议"部分
  → AI 分析每个计划的状态
  → 提供可操作的建议
```

### 5. 进度条生成

```
定义进度条字符：
  → 已完成：█
  → 未完成：░
  → 总格数：10

生成逻辑：
  percentage = (已完成 / 总数) * 100
  filled = round(percentage / 10)
  empty = 10 - filled
  进度条 = █ repeated {filled} + ░ repeated {empty}

示例：
  0%:   ░░░░░░░░░
  23%:  ██░░░░░░░░
  50%:  ██████░░░░
  75%:  ████████░░
  100%: ██████████

边界情况：
  → 总数为 0：░░░░░░░░░ (0%)
  → 已完成 > 总数：██████████ (100%)
```

### 6. 下一步建议（可选）

```
IF 用户没有指定筛选且计划数量 > 0:
  → AI 分析每个计划的状态
  → 生成可操作的建议

分析维度：
  1. 长时间未更新的计划（> 7 天）
  2. 进度过半但长期停滞的计划
  3. 刚创建的建议优先完成的计划
  4. 有大量待执行任务的计划

建议格式：
  "下一步建议：
    • {计划名} 有 {N} 个待执行任务，建议优先完成 {模块名} 部分
    • {计划名} 进度过半，建议检查是否有阻塞任务
    • {计划名} 刚创建，建议先完成技能定义"
```

## 技术细节

### 数据解析

**spec.md 解析**：
```markdown
# 标题 → 提取标题

## 元数据
- 创建时间：2026-04-24 → 提取创建时间
- 状态：active → 提取状态
- 相关 Wiki：current/architecture.md → 提取相关 Wiki

## 目标
{目标内容} → 提取简短描述（前 50 字）
```

**tasks.md 解析**：
```
匹配规则：
  `- [` 开头的行 → 任务
  `- [x]` → 已完成
  `- [ ]` → 待执行
  `- [-]` → 取消（不计入总数）
  `- [?]` → 阻塞（计入待执行）

统计：
  总数 = 已完成 + 待执行
  进度 = (已完成 / 总数) * 100
```

### 自然语言解析

**支持的模式**：
1. 包含关键词："包含"、"含有"、"关于"
2. 进度筛选："超过"、"大于"、"低于"、"少于"
3. 时间筛选："今天"、"昨天"、"本周"
4. 状态筛选："状态为"、"状态是"

**示例**：
- "显示包含 website 的计划" → 关键词 = "website"
- "进度超过 50% 的计划" → 进度 > 50%
- "昨天创建的计划" → 创建时间 = 昨天
- "状态为 active 的计划" → 状态 = "active"

### 状态值映射

```
spec.md 中的状态 → 显示标签

active → "active"（活跃开发中）
draft → "draft"（草稿，刚创建）
paused → "paused"（暂停）
blocked → "blocked"（阻塞）
completed → "completed"（已完成，但未归档）
```

## 注意事项

### 只读约束

**允许的操作**：
- ✅ 读取 `.wikiflow/config.json`
- ✅ 读取 `docs/changes/active/` 目录
- ✅ 解析 `spec.md`、`tasks.md`、`decisions.md`
- ✅ 显示格式化的输出

**禁止的操作**：
- ❌ 修改任何文件
- ❌ 创建或删除文件
- ❌ 执行 Git 操作
- ❌ 修改代码仓库

### 错误处理

**文件缺失**：
```
⚠️ 警告：计划 2026-04-24-xxx 缺少 spec.md，跳过
```

**目录为空**：
```
📭 当前没有活跃的开发计划

提示：
  → 使用 /wf-plan 创建新计划
  → 或检查 {paths.changes}/active/ 目录
```

**解析失败**：
```
❌ 错误：无法解析 tasks.md

文件路径：docs/changes/active/2026-04-24-xxx/tasks.md

可能原因：
  • 文件格式不正确
  • 缺少任务标记（- [ ] 或 - [x]）

建议：
  → 检查文件格式
  → 或使用 /wf-fix 修复
```

## 最佳实践

### 使用场景

1. **每日查看**：
   ```bash
   /wf-status
   ```
   快速了解所有计划的进度

2. **决策前**：
   ```bash
   /wf-status --active
   ```
   查看活跃计划，决定下一步工作

3. **详细分析**：
   ```bash
   /wf-status <计划名> --detail
   ```
   查看具体计划的完整任务列表

4. **问题排查**：
   ```bash
   /wf-status --blocked
   ```
   查找阻塞的计划

### 与手动查询的区别

| 维度 | 手动查询 | /wf-status |
|------|---------|-----------|
| 输出格式 | 不一致 | 标准化 |
| 信息完整度 | 依赖 AI 提问 | 完整展示 |
| 可操作性 | 需要多次查询 | 一次获取全部 |
| 进度可视化 | 无 | ASCII 进度条 |
| 筛选功能 | 需要重新查询 | 参数化筛选 |
