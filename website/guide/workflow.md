# 工作流说明

WikiFlow 的工作流围绕"文档先行"的理念设计，确保每次开发都有清晰的规划和完整的记录。

## 完整工作流

```
┌─────────────────────────────────────────────────────────────┐
│                        开发前准备                            │
└─────────────────────────────────────────────────────────────┘
                            │
                    ┌───────▼───────┐
                    │   wf-search   │ ← 查询现有文档，了解背景
                    └───────┬───────┘

                    ┌───────▼───────┐
                    │    wf-plan    │ ← 制定开发计划
                    └───────┬───────┘
                            │
                            ▼
                    ┌───────────────┐
                    │  生成 spec.md  │ ← 技术方案
                    │ 生成 tasks.md  │ ← 任务清单
                    └───────┬───────┘
                            │
┌───────────────────────────▼───────────────────────────────────┐
│                        开发执行                                │
└─────────────────────────────────────────────────────────────┘
                            │
                    ┌───────▼───────┐
                    │     wf-do     │ ← 逐个执行任务
                    └───────┬───────┘
                            │
                    ┌───────▼───────┐
                    │    发现问题？  │
                    └───────┬───────┘

                   ┌─▼──┐    │    │
                   │wf-fix│   │    │
                   └─┬──┘   │    │
                     └──────┼────┘

                    │  所有任务完成？│
                    └───────┬───────┘

                    └───────────────┘
```

## 各阶段详解

### 1. 开发前准备

**目标**：确保开发前充分了解背景

```bash
# 步骤 1：查询现有文档
/wf-search "用户认证"

# 步骤 2：制定开发计划
/wf-plan "添加注册功能"

# 步骤 3：确认方案
# WikiFlow 会生成 spec.md，确认技术方案
```

**产出**：
- `docs/changes/active/{slug}/spec.md` - 技术方案
- `docs/changes/active/{slug}/tasks.md` - 任务清单

### 2. 开发执行

**目标**：按计划执行任务

```bash
# 开始执行
/wf-do

# WikiFlow 会：
# - 逐个执行 tasks.md 中的任务
# - 实时更新任务状态
# - 显示开发进度
```

**遇到问题时**：
```bash
# 使用 wf-fix 修正
/wf-fix "注册成功后应该自动登录"

# WikiFlow 会：
# - 更新 spec.md
# - 添加新任务到 tasks.md
# - 记录决策日志
```

**产出**：
- 完成的代码变更
- 更新的 tasks.md（标记已完成）

### 3. 完成变更

**目标**：将变更同步到 Wiki

```bash
# 完成开发
/wf-finish

# WikiFlow 会：
# - 检查文档质量（lint）
# - 更新 Wiki 知识库
# - 归档 active 变更
```

**产出**：
- `docs/wiki/current/` - 更新或新增文档
- `docs/wiki/history/` - 旧版本归档
- `docs/wiki/decisions/` - 决策记录
- `docs/changes/archive/` - 已归档的变更

## 最佳实践

### 1. 文档先行的价值

**为什么先写文档？**

- ✅ **思路清晰**：写文档的过程就是梳理思路的过程
- ✅ **设计完整**：提前发现问题和边界情况
- ✅ **沟通顺畅**：团队成员可以提前review
- ✅ **有据可查**：后续维护时可以了解设计初衷

### 2. 小步快跑

**推荐做法**：

```bash
# ✅ 推荐：小功能独立开发
/wf-plan "添加注册功能"
/wf-do
/wf-finish

/wf-plan "添加密码重置功能"
/wf-do
/wf-finish

# ❌ 不推荐：一次开发太多功能
/wf-plan "重构整个认证系统"  # 太大，难以review
```

### 3. 保持同步

**开发过程中**：

- ✅ 每完成一个任务，立即更新 tasks.md
- ✅ 发现设计问题，立即用 wf-fix 修正
- ✅ 不要拖延 wf-finish，及时同步到 Wiki

### 4. 定期维护

**建议频率**：

```bash
# 每周检查一次文档质量
/wf-lint

# 每月更新一次 WikiFlow
/wf-update

# 每个 Sprint 结束后，review 变更记录
cat docs/wiki/log.md
```

## 常见场景

### 场景 1：新功能开发

```bash
wf-search "相关功能"      # 了解背景
wf-plan "新功能"          # 制定计划
wf-do                     # 执行开发
wf-finish                 # 完成变更
```

### 场景 2：Bug 修复

```bash
wf-plan "修复 XXX bug"    # 制定修复计划
wf-do                     # 执行修复
wf-finish                 # 更新文档
```

### 场景 3：需求改进

```bash
wf-plan "优化 XXX 流程"   # 制定改进计划
wf-do                     # 执行改进
wf-finish                 # 归档旧版本
```

### 场景 4：发现新问题

```bash
wf-do                     # 开发中
wf-fix "发现新问题"       # 修正计划
wf-do                     # 继续执行
wf-finish                 # 完成变更
```

## 工作流图解

### 文档流转

```
raw/ (原始文档)
  ↓
  整理到 Wiki
  ↓
current/ (当前实现) ← 开发时查询这里
  ↓
  发生变更
  ↓
history/ (历史版本) ← 旧版本归档到这里
```

### 变更管理

```
changes/active/ (开发中)
  ↓ wf-finish
changes/archive/ (已归档)

同时生成：
  - wiki/current/ (更新)
  - wiki/history/ (归档)
  - wiki/decisions/ (决策)
```
