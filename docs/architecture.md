# 串行任务执行模型

本文档详细介绍 WikiFlow 采用的串行任务执行模型，解释其设计原理和实际应用。

## 什么是串行任务执行？

WikiFlow 采用**串行任务队列**的方式处理开发任务：

```
任务队列：[任务 A, 任务 B, 任务 C]

执行流程：
1. 取出任务 A → 执行 → 完成
2. 取出任务 B → 执行 → 完成
3. 取出任务 C → 执行 → 完成
```

**核心特征**：
- 同一时间只有一个任务在执行
- 任务按顺序依次完成，不并行处理
- 共享代码仓库和基础设施
- 简单的状态管理和上下文切换

## 为什么选择串行模式？

### AI 时代的开发特点

在 AI 辅助开发的场景下，任务执行呈现出新的特征：

**任务时间短**：
- AI 完成单个任务通常只需几分钟到十几分钟
- 相比人工开发的小时级耗时，速度大幅提升

**人机交互是瓶颈**：
- 人类描述需求、审查结果需要时间
- AI 执行代码的速度远快于人类输入的速度
- 等待人类反馈是主要的耗时环节

**并发收益有限**：
- 由于人机交互的串行特性，理论最大并发量接近 1.0
- 即使尝试并行，实际节省的时间也非常有限

### 串行模式的优势

**1. 避免数据迁移冲突**

当多个任务同时修改数据库结构时：
- 串行执行：任务 A 完成字段变更后，任务 B 基于新结构继续
- 并行执行：任务 A 废弃字段，任务 B 仍在使用，导致全局报错

**2. 简化基础设施依赖**

不需要为每个任务隔离：
- 数据库实例
- Redis 缓存
- 服务端口
- 环境变量

所有任务共享同一套基础设施，配置简单，维护成本低。

**3. 前后端统一管理**

对于包含前后端的项目：
- 不需要使用 Git Worktree 进行复杂隔离
- 不需要管理多个工作目录
- 分支切换简单清晰

**4. 尊重任务间的依赖关系**

实际开发中，任务往往存在依赖：
- 任务 B 可能需要任务 A 创建的 API 接口
- 任务 C 可能依赖任务 B 定义的数据库表
- 串行执行自然满足这些依赖关系

**5. 适合目标用户群体**

WikiFlow 面向的用户场景：
- 个人开发者项目
- 小型团队（1-5 人）
- MVP 快速验证
- 原型开发

这些场景通常不需要复杂的并行优化，简单可靠更重要。

## 串行执行的实现方式

### 基本流程

```
开始
  ↓
检查是否有活跃任务
  ↓
有 → 切换到任务分支
  ↓
执行任务（读取 spec.md 和 tasks.md）
  ↓
AI 根据任务要求修改代码
  ↓
完成任务，切回主分支
  ↓
更新任务状态
  ↓
检查下一个任务
  ↓
重复上述流程
```

### 状态管理

串行模式下的状态非常简单：

**当前状态**：
- 正在执行的任务 ID
- 任务所在的 Git 分支
- 任务的执行进度

**无需管理**：
- 多任务之间的资源竞争
- 并发锁机制
- 复杂的任务调度算法

### 错误处理

串行模式的错误处理更简单：

- 任务失败时，可以立即发现问题
- 不会因其他任务的影响而产生连锁反应
- 回滚操作简单：直接切换到主分支即可
- 调试方便：只需要关注当前任务的上下文

## 与并行模式的对比

| 维度 | 串行模式 | 并行模式 |
|------|---------|----------|
| **复杂度** | 低，易于理解和维护 | 高，需要复杂的隔离机制 |
| **基础设施** | 共享，配置简单 | 需要隔离，配置复杂 |
| **数据一致性** | 天然保证，无冲突 | 需要额外的同步机制 |
| **调试难度** | 低，上下文清晰 | 高，需要考虑并发问题 |
| **适用场景** | 小型项目、快速迭代 | 大型系统、大规模团队 |
| **实际收益** | 稳定可靠 | 理论上更快，但收益有限 |

## 实际影响

串行任务执行模型影响了 WikiFlow 的多个方面：

### 工作流程设计

- `/wf-do` 技能按顺序执行 tasks.md 中的任务
- 每个任务完成后才进入下一个任务
- 保持清晰的执行轨迹和可追溯性

### 文档组织

- `docs/changes/active/` 目录中同时只活跃一个变更集
- spec.md 和 tasks.md 描述的是当前正在执行的任务
- 历史变更记录在 decisions/ 和 history/ 中

### 工具简化

- 不需要复杂的任务调度器
- 不需要工作流隔离机制
- 不需要并发性保护代码

## 常见疑问

### Q: 串行执行会不会很慢？

A: 实际上不会。因为：
- AI 执行单个任务很快（几分钟）
- 人机交互是串行的，无法真正并行
- 串行避免了并行带来的额外开销
- 总体效率反而更高

### Q: 如果有紧急任务怎么办？

A: 可以中断当前任务：
- 保存当前任务状态
- 切换到新任务执行
- 完成后恢复之前的任务
- 这种手动干预比自动并行更可控

### Q: 大型项目能用串行模式吗？

A: WikiFlow 的定位是小型项目和快速迭代：
- 如果项目规模很大，可能需要考虑其他方案
- 但对于大多数个人和小团队项目，串行模式完全够用
- 简单可靠比理论上的性能更重要

## 总结

串行任务执行模型是 WikiFlow 的核心设计决策之一，它基于对 AI 时代开发特点的深入理解：

- **简单优于复杂**：减少不必要的复杂性
- **可靠优于理论最优**：实际可用的方案胜过理论完美
- **速度源于简洁**：没有额外开销的执行更高效

这种设计让 WikiFlow 能够专注于核心价值：通过文档驱动的方式，让人类和 AI 高效协作，快速完成开发任务。
